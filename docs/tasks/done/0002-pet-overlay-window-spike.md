# Task: Spike — transparent click-through pet overlay window on all 3 OS

## Context

Highest technical risk in the project ([[../../architecture|Architecture]] §Important Constraints): a transparent, borderless, always-on-top window that is click-through everywhere *except* over the pet sprite, on Windows, macOS, and Ubuntu (X11 + Wayland assessment).

## Goal

Prove (or disprove) the overlay approach per OS and document per-platform findings before building on it.

## Scope

In scope: transparent window flags, `set_ignore_cursor_events` toggling driven by canvas hit-testing on mouse move, multi-monitor positioning, skip-taskbar, a moving test square. Wayland: document what works per compositor (GNOME/KDE) and specify the fallback (docked corner window).

Out of scope: real sprites, behavior AI, economy.

## Acceptance Criteria

- [ ] Test square is visible over other apps, draggable, and clicks pass through empty regions on Win/macOS/X11. **Implemented, not hands-on tested** - no display/GUI in this sandbox and no Windows/macOS/X11 machine to test on. Needs a human on each OS.
- [x] Findings + Wayland fallback decision written to `docs/knowledge/overlay-platform-notes.md`.
- [ ] CPU <1% while square is idle. **Implemented (30fps active / 2fps idle tick), not measured.** Needs a human to actually watch Activity Monitor/Task Manager while the idle phase runs.

## Implementation Notes

- `src-tauri/tauri.conf.json`: added a second window, `"overlay"` - transparent, `decorations: false`, `alwaysOnTop`, `skipTaskbar`, `resizable: false`, `focus: false` (so spawning it doesn't steal focus from other apps), `acceptFirstMouse: true` (macOS: don't swallow the first click while unfocused). `beforeDevCommand`/`beforeBuildCommand` now build `ui/overlay` before `ui/dashboard`.
- `src-tauri/capabilities/overlay.json`: new capability scoped to the `overlay` window granting the window-control permissions it needs (`allow-set-ignore-cursor-events`, `allow-start-dragging`, `allow-set-position`/`-size`, monitor queries). **Not yet validated against `gen/schemas/capabilities.json`** (only generated by a real `cargo build`) - if any permission identifier is wrong, Tauri's ACL validation will fail loudly and name the bad one; that's a fast, self-correcting failure if so.
- `src-tauri/src/overlay_window.rs`: new module, `fit_to_primary_monitor()` - queries `window.primary_monitor()` and resizes/repositions the overlay to cover it at startup; called from `lib.rs` `.setup()`. Logs and no-ops on failure rather than panicking (spike-appropriate: wrong size shouldn't crash the app).
- `ui/overlay/` (new, standalone npm package per `docs/architecture.md`'s planned layout - vanilla TypeScript, no framework, per [[../../decisions/0003-canvas-sprite-rendering|ADR-0003]]):
  - `src/main.ts`: bouncing test square; `mousemove` hit-tests the cursor against the square (with a small grab-padding) and calls `Window.setIgnoreCursorEvents()` only when the hit state flips (avoids spamming the IPC boundary on every mouse pixel); `mousedown` over the square calls `Window.startDragging()`; alternates every ~6s between an "active" tick (~30fps, bouncing) and an "idle" tick (~2fps, stationary) to exercise both tick rates from `docs/architecture.md` §Runtime and Deployment.
  - Built with `esbuild` (bundled to `ui/dashboard/static/overlay/overlay.js`) and a plain `cp` of `index.html` alongside it - SvelteKit's adapter-static copies `static/` verbatim, so this rides the existing single `frontendDist` without needing a second dev server or a multi-entry bundler config (lower-risk choice given nothing here could be compiler-verified for the Rust side anyway; see Verification Results for what *was* actually run and checked).
  - `docs/knowledge/overlay-platform-notes.md`: per-OS notes and the Wayland fallback decision (docked corner window when `WAYLAND_DISPLAY`/`XDG_SESSION_TYPE=wayland` is detected - detection itself not wired up yet, that's explicit follow-up work once the primary approach is confirmed to fail on a real Wayland session).
- `.github/workflows/ci.yml`: `build` job now also runs `npm ci --prefix ui/overlay` (needed by `beforeDevCommand`/`beforeBuildCommand`).
- `.gitignore`: added `/ui/dashboard/static/overlay/` (generated by `ui/overlay`'s build script - not source) and `/ui/overlay/node_modules/`.

**Round 2 (user ran `cargo tauri dev` for real - a real bug found):** `beforeDevCommand` failed immediately: `npm --prefix ../ui/overlay run build` tried to open `/Users/taio/Desktop/projects/ui/overlay/package.json` - one directory too far up, missing the `tokengochi` project folder entirely. **Round 3 (user ran it on macOS - another real bug found):** overlay window rendered with an opaque white background instead of transparent, even though `"transparent": true` was set on the window and `ui/overlay/index.html` already had `background: transparent` on `html`/`body`. Root cause: on macOS, Tauri v2's WKWebView-backed windows require `app.macOSPrivateApi: true` in `tauri.conf.json` for window transparency to take effect at all - without it, the private APIs needed to make the WKWebView layer transparent aren't used, so it falls back to an opaque default regardless of the `transparent` window flag or the page's own CSS. Fixed by adding `"macOSPrivateApi": true` under `"app"` in `src-tauri/tauri.conf.json`. Documented in [[../../knowledge/overlay-platform-notes|Overlay Platform Notes]] §macOS.

**Still needed:** please re-run `cargo tauri dev` on macOS to confirm the background is now transparent, then continue through the rest of the unverified acceptance criteria (click-through, drag, idle CPU).

**Round 4 (user confirmed transparency fixed; found click-through does not work):** background now renders transparent, but clicks never pass through to apps below, anywhere on screen. Root cause in `ui/overlay/src/main.ts`: `updateHitTest()` only calls `setIgnoreCursorEvents` when the computed `hit` state differs from `lastHit`, and `lastHit` was initialized to `false` as a bare JS assumption that the window *already* starts in the click-through (ignoring) state. It doesn't - a Tauri window's native `ignoreCursorEvents` defaults to `false` (capturing everything) until explicitly set via the API, and nothing sets it at startup. So if the cursor's first `mousemove` lands off the square (the common case), `hit === lastHit` (`false === false`) and the function returns early without ever calling the API - the window stays in the native default (capturing) indefinitely, until the cursor happens to cross the square once and leave again (two flips), which is the only path that produces an actual call. Fixed by adding an explicit `void appWindow.setIgnoreCursorEvents(true)` call right after startup in `main.ts`, so the native state is forced into sync with the assumed `lastHit = false` instead of relying on it matching by coincidence. Rebuilt via `npm run build` in `ui/overlay`.

**Still needed:** re-run `cargo tauri dev` on macOS to confirm click-through now works from the start (not just after first touching the square), then confirm dragging and idle CPU.
